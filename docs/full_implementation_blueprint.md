# 小说写作工作流：全量开发蓝图 (Full Blueprint)

本建设方案将 `novel-splitter` 升级为具备“市场嗅觉”与“拆书深度”的工业化网文分析系统。

## 一、 系统架构目标

1.  **后台常驻**：程序随开机自启或手动开启后，最小化到系统托盘，不影响日常使用。
2.  **定时调度**：每天凌晨或指定时间点，自动启动全自动化流。
3.  **增量全量分析**：对榜单全量 30-100 本书进行画像；已分析过的保持快照，新书强制触发深度拆解。
4.  **深度报告**：产出一份融合宏观题材趋势与微观爽点拆解的每日 Markdown 报告。

---

## 二、 核心模块实现逻辑

### 1. 基础保活模块 (Infrastructure)
- **托盘图标 (System Tray)**：
    - 拦截 `CloseRequested` 窗口事件，通过 `window.hide()` 让程序在后台保持活跃。
    - 增加托盘菜单：`显示主界面`、`立即全量扫榜 (强制)`、`设置定时时间`、`退出应用`。
- **调度器 (Tokio Scheduler)**：
    - 在 `tauri::run` 的 `setup` 钩子中启动后台 `Loop`。
    - 每 60 秒检查一次当前时间点。若命中设定时间且今日未执行，则串联后续逻辑。

### 2. 多维度增量分析引擎 (Analysis Engine)
- **数据存储 (Persistence)**：
    - 目录：`novel-splitter/analysis_data/`。
    - `history_summary.json`：记录每一本书的 `book_id`, `rank_history`, `metadata` 以及其 `ai_deconstruct_report` 缓存。
- **黑马算法 (Velocity)**：
    - 读取今日抓取的 `rank[30]` vs 昨日 `rank[30]` 数据。
    - 若 `diff > 50` 或 `new_arrival == true`，在报告中标记为 **“高关注书籍”**。
- **去重/缓存策略 (Caching)**：
    - 若书籍已在库中且“前三章已拆解”：直接从缓存中调取它的深度分析数据编入今日总报，**不重复调用 AI**。
    - 仅对“新面孔”或指定的“深度扫描目标”发起正文下载与 AI 请求。

### 3. 微观深度正文扫描 (Deep Decomposition)
- **采样模式**：自动下载目标书籍的前 3-5 章（黄金三章）。
- **精修 Prompt (指令集)**：
    - **核心维度**：开篇冲突点定位、金手指底层逻辑及其代价、情绪节奏曲线（压抑/释放占比）、章末勾子类型（悬念 or 期待感）。
- **并发控制**：下载串行（防封 IP），AI 请求并行（快速回传）。
