# UI 评审与高真还原工作流

更新时间：2026-05-16

## 背景

这次 `novel-splitter` 的 UI 优化过程中，出现过一个典型问题：

- 已经参考了高真稿
- 也已经改了代码
- 但**没有先拿到真实运行页面截图**
- 结果导致“代码看起来像是同风格”，但**实际页面并没有贴近高真稿**

这说明：

> **看代码 + 凭感觉调整 ≠ 已完成高真还原**

因此，后续凡是涉及 UI 优化、高真还原、页面定稿，都必须遵循下面这套流程。

---

## 一、适用范围

以下任务默认必须使用本工作流：

- UI 优化
- 页面排版调整
- 高真稿还原
- 视觉统一
- 页面定稿
- 交互状态复核

---

## 二、核心原则

### 1. 没有真实截图，不算完成视觉对照

如果没有拿到：

- 当前实际运行页面截图
- 高真稿参考图
- 两者的差异结论

就不能宣称：

- “已经贴稿”
- “已经定稿”
- “视觉已对齐”

### 2. 必须先对照，再改代码

正确顺序是：

1. 找参考图
2. 截实际运行图
3. 做差异清单
4. 再改代码
5. 再截图复核

而不是：

1. 看代码
2. 凭感觉改
3. 主观判断“差不多”

### 3. 二次截图复核是必做项

即使代码已经改完并通过构建，仍然需要：

- 再次截图
- 再次和高真稿对照
- 再判断是否可以叫“定稿”

---

## 三、标准执行流程

## Step 1：确认视觉基线

先明确：

- 高真稿路径
- 当前要对齐的页面
- 当前页面对应的状态（空态 / 列表态 / 详情态 / 弹窗态等）

示例：

- `output/ui-高真/library_v2.png`
- `output/ui-高真/novel-detail_v1.png`

---

## Step 2：必须拿到真实运行页面截图

### 推荐路径

优先使用本地预览 + 浏览器截图：

1. 启动页面
   - 如：`npm run preview`
   - 或其他真实可运行的本地服务

2. 打开真实页面
   - 本地地址示例：`http://127.0.0.1:4173`

3. 截图保存到项目内
   - 建议路径：`output/ui-高真/actual-*.png`

### 本次验证可行的方法

本项目本次已验证过的一条可行路径：

- 启动 `vite preview`
- 使用 **headless Chrome** 对本地地址截图
- 输出文件：
  - `output/ui-高真/actual-library-page.png`

### 约束

如果当前环境拿不到截图，必须明确说明：

> “当前还没有拿到真实运行页面截图，因此还不能判断是否贴稿。”

不能跳过这一步直接给“定稿”结论。

---

## Step 3：建立差异清单

对照：

- 实际截图
- 高真稿

至少检查以下维度：

### 布局
- 区块顺序是否一致
- 左右分栏是否一致
- 头部 / 主体 / 底栏是否一致

### 尺寸与留白
- 顶栏高度
- 区块间距
- 内容面板边距
- 空状态区域占比

### 组件样式
- 按钮大小与主次层级
- 标签/筛选器样式
- 卡片圆角、边框、阴影
- 图标风格

### 文本层级
- 标题字号
- 副标题字号
- 辅助文案颜色
- 字重是否一致

### 状态表现
- 空状态
- 加载态
- 选中态
- 报告详情态
- 弹窗态

---

## Step 4：按差异清单改代码

这一步必须以“差异项”为输入，而不是“凭感觉美化”。

每一轮改动都应该回答：

- 这次改的是哪一个视觉差异？
- 改动是否能缩小与高真的距离？
- 有没有引入新的偏差？

---

## Step 5：构建验证

代码改动后，至少执行：

- `npm run build`

如有需要，再执行页面专属验证。

---

## Step 6：二次截图复核

改完后必须再次截图。

对照结果应明确分成：

- 已对齐
- 基本对齐但仍有差异
- 尚未对齐

只有在“二次截图复核”之后，才可以说“接近定稿”或“可以定稿”。

---

## 四、交付标准

一个合格的 UI 定稿交付，至少应包含：

1. 高真稿参考图路径
2. 当前实际截图路径
3. 改动文件列表
4. 差异清单 / 修正说明
5. 构建验证结果
6. 仍未对齐的风险点（如果有）

---

## 五、建议命名规范

### 参考图
- `output/ui-高真/library_v2.png`
- `output/ui-高真/novel-detail_v1.png`

### 实际截图
- `output/ui-高真/actual-library-page.png`
- `output/ui-高真/actual-novel-detail-page.png`
- `output/ui-高真/actual-report-page.png`

### 差异说明 / 规格
- `output/ui-高真/ui-final-spec-YYYY-MM-DD.md`
- `docs/ui-review-workflow.md`

---

## 六、本项目本次经验结论

本次经验可以概括为一句话：

> **UI 任务里，没有真实运行截图，就不要宣布“贴稿完成”。**

后续在 `novel-splitter` 中处理任何 UI 定稿任务，都默认采用本工作流。
