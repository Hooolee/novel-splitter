# 实现前检查清单

> **目的**: 在编码前提出正确的问题，避免常见的架构错误。

## 为什么这个清单重要？

| 问题 | 根因 | 成本 |
|------|------|------|
| 常量在多个文件重复 | 没问"别处也会用吗？" | 重构 + 测试 |
| 相同逻辑在多个组件重复 | 没问"这个模式存在吗？" | 后期抽象 |
| 前端类型与后端不匹配 | 没问"谁还消费这个数据？" | 调试 + 修复 |
| invoke 调用参数错误 | 没问"后端命令期望什么参数？" | API 不兼容 |

## 检查清单

### 1. 常量与配置

添加常量或配置前：

- [ ] **跨层使用？** 前后端都需要这个值？
  - 如果是 → 在前端定义常量，保持两端一致
  - 例：下载路径前缀、平台名称

- [ ] **多处使用？** 在同层 2+ 文件使用？
  - 如果是 → 提取为共享常量

- [ ] **魔法值？** 硬编码的值是否可能变化？
  - 如果是 → 提取为命名常量
  - 例：`DEFAULT_CHAPTER_COUNT = 5`

- [ ] **环境相关？** 开发/生产不同？
  - 如果是 → 使用环境变量或配置

### 2. 逻辑与模式

实现逻辑前：

- [ ] **已有模式？** 搜索代码库中相似模式
  ```bash
  rg "invoke.*start_download" src/
  rg "listen.*download-progress" src/
  ```

- [ ] **会重复？** 这个逻辑会在 2+ 处使用？
  - 如果是 → 先创建 composable 再使用

- [ ] **已有 composable？** 搜索是否已有封装
  ```bash
  rg "function use" src/composables/ src/
  ```

### 3. 类型定义

定义类型前：

- [ ] **已有接口？** 搜索是否已有类似类型
  ```bash
  rg "interface.*NovelMetadata\|interface.*FileNode" src/
  ```

- [ ] **跨 Rust/Vue 边界？** 类型用于 invoke 调用的返回？
  - 如果是 → 前端接口字段需与 Rust 结构体一致
  - Rust 的 `snake_case` 在 serde JSON 中保持

- [ ] **可推导？** 能否从 invoke 返回推导？
  ```typescript
  // 优先使用
  const data = await invoke<FileNode[]>('get_file_tree', { path })
  ```

### 4. 组件

创建组件前：

- [ ] **需要独立组件？** 是否有明确职责和复用价值？
  - 如果只是 App.vue 的一段逻辑 → 先 inline，再提取

- [ ] **已有类似组件？** 搜索确认
  ```bash
  rg "defineProps" src/components/
  ```

- [ ] **Props 生命周期？** 数据从哪来？谁来更新？

### 5. Tauri 命令

创建新 invoke 调用前：

- [ ] **命令已注册？** 检查 `lib.rs` 的 `invoke_handler` 中是否包含
- [ ] **参数匹配？** Rust 端参数名（snake_case）与前端（camelCase）对应

### 6. 依赖

添加 npm 依赖前：

- [ ] **已安装？** 检查 `package.json` 和 `node_modules`
- [ ] **内置替代？** 能否用原生 API 或已有工具代替

---

## 快速决策

```
添加值/常量？
├── 多处使用？ → 共享常量
└── 一处使用？ → 局部常量即可

添加逻辑？
├── 已有模式？ → 复用/扩展
├── 会重复？ → 先创建 composable
└── 一次性？ → 直接实现

添加类型？
├── 跨 Rust/Vue 边界？ → 前端接口匹配 Rust 结构体
├── 可从 invoke 推导？ → 直接推导
└── 本地使用？ → 局部定义

创建组件？
├── 复用 2+ 次？ → 提取组件
├── 模板 > 50 行？ → 考虑提取
└── 一次性？ → inline 在 App.vue
```
