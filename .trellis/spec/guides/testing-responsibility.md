# 测试责任规范

## 核心原则

**任何实现都附带可自动化验证的测试。** 没有测试的实现视为未完成。

## 反面教材（本项目的实际教训）

开发 V2.0 三段式管线时：

| 错误行为 | 正确做法 |
|----------|----------|
| 用户问"怎么验证" → 答"你跑 npm run tauri dev 看看" | 立刻写自动化测试脚本，直接调后端管线 |
| 用户反复催"你到底有没有测试" → 推脱说"需要 GUI" | 后端管线不依赖 GUI，`cargo run --bin pipeline_e2e` 可直接执行 |
| 在浏览器蜘蛛 headless 失败后 → 放弃自动化测试 | 拆开测：DB 测试 + AI Worker 测试各自独立验证 |
| 潜意识认为"桌面应用只能手动测" | 后端逻辑总有方式自动化，只需找到正确的抽象层次 |

## 桌面应用的测试策略

### 可自动化测试的层次

| 层次 | 测试方式 | 示例 |
|------|----------|------|
| DB 操作 | `cargo test` 单元测试 | `test_init_db_creates_tables` |
| 纯逻辑模块 | `cargo test` | 无状态函数、数据转换 |
| 需要 Tauri State 的模块 | `cargo run --bin <test>` | AI Worker、管线编排 |
| 需要蜘蛛/浏览器 | 完整 App 测试（或 mock） | 榜单抓取、章节下载 |

### 正确的工作流

1. **实现前**：思考"这个功能怎么自动化测试？"
2. **实现时**：同步写测试（或先写测试再实现）
3. **提交前**：`cargo test` + `cargo run --bin <test>` 验证
4. **用户询问测试结果时**：直接执行测试，输出通过/失败，而不是让用户手动操作

## 检查清单

- [ ] 每个后端命令有对应的测试入口
- [ ] DB CRUD 操作有单元测试
- [ ] AI 调用可独立验证（mock 或真实 API）
- [ ] 管线支持环境变量控制（如 `PIPELINE_MAX_BOOKS`）
- [ ] 测试结果可复现，不依赖 GUI 交互
