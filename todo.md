## 🛑 核心设计哲学
将“手动拆单本书”的死板工具，升级为**“高性能、强结构化、多维透视”**的网文市场情报看板。利用 SQLite 数据库作为状态机，驱动异步流水线。

## 🚀 第二部分：后端核心攻坚（待改造）

### 任务一：任务驱动型异步流水线 (Task-Driven Pipeline)
> **目标**：彻底解决“一本书死掉卡死全盘”和“Token消耗重复”的问题。

*   [ ] **1. 扫榜分发 (Producer)**：
    *   浏览器蜘蛛只负责飞速抓取榜单 Top 30 的 `book_id` 和书名。
    *   写入 `Novels`（基础信息）和 `Rank_History`（当期名次）。
*   [ ] **2. 抓取 Worker (并发数 3)**：
    *   **触发**：扫描数据库中“有 Novel 记录但无 Chapter 内容”的书。
    *   **动作**：使用 `tokio::sync::Semaphore(3)` 并发拉取前三章正文，存入 `Chapters` 表。
    *   **容错**：单本书报错则跳过，下次扫榜自动重试补齐。
*   [ ] **3. AI 提纯 Worker (并发数 3)**：
    *   **触发**：扫描数据库中“有 Chapter 内容但无 AI 评价”的书。
    *   **动作**：并发调用 LLM。
    *   **改动（重要）**：Prompt 升级，不再输出大段 Markdown 描述，强制要求输出 `[事件, 目的, 情绪, 爽点]` 的 **JSON 细纲**，存入 `Chapters` 表的 `outline_json` 字段。

### 任务二：多视角智能体深度评估 (AI Multi-Agent Debate)
> **目标**：在章纲提纯后，进行深度的商业价值推演。

*   [ ] **1. 三重 Agent 模拟**：
    *   **毒舌读者 Agent**：抓出爽点、槽点、毒点。
    *   **资深主编 Agent**：评估商业价值、金手指、市场契合度。
    *   **白金作者 Agent**：分析起承转合、伏笔与悬念。
*   [ ] **2. 评分组装**：
    *   汇总三个 Agent 的打分（1-10分），整合成 JSON 存入 `Novels` 表的 `ai_reviews_json` 字段。
