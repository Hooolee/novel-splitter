## 各任务详情

### 任务一：三段式异步流水线

将原有的单体循环 `run_full_analysis_pipeline` 拆为三段：

```
Phase 1: Producer（扫榜分发）
  └─ 只抓 book_id + title + URL，存入 novels 表
Phase 2: Fetch Workers（并发抓取）
  └─ Semaphore(3) 并发抓前 3 章，存入 chapters 表
Phase 3: AI Workers（并发提纯）
  └─ Semaphore(3) 并发 Call LLM，输出 JSON 细纲存入 chapters.outline_json
```

**关键决策**：并发数硬编码 3（Semaphore），可环境变量覆盖 `PIPELINE_MAX_BOOKS`。

### 任务二：多 Agent 分歧驱动评估

新增 Phase 4，每本小说并行调用 3 个 AI Agent（毒舌读者/资深主编/白金作者），输出分歧驱动的评估 JSON：

```json
{
  "agents": {
    "reader":  { "vote": "yes|no|maybe", "focus": ["文笔", "节奏"], "comment": "..." },
    "editor":  { "vote": "...", "focus": [...], "comment": "..." },
    "author":  { "vote": "...", "focus": [...], "comment": "..." }
  },
  "consensus": "all_yes | majority_yes | divergent | majority_no | all_no",
  "meta": { "model": "...", "generated_at": "...", "input_chapters": 3 }
}
```

**关键决策**：
- 分歧驱动（三路 `tokio::join!` 并发调用），保留分歧信息
- 单 Agent 失败记 `null` 不影响其余，三 Agent 全失败跳过该书
- TODO/多 Agent 不覆盖旧值（保留上一次评估）

### 任务三：前端 UI 瘦身

砍掉 V1「下载 Tab」，**单本下载也走新管线**（不再写文件），信息流统一到 SQLite。

**关键决策**：
| # | 决策 |
|---|---|
| D1 | 下载 Tab 完全砍掉 |
| D2 | 单本入口 → library Tab「+ 添加书籍」模态框 |
| D3 | 手动扫榜 → 报告 Tab「立即扫榜」按钮 |
| D4 | 单本走新管线（`trigger_full_scan(url, platform)`） |
| D5 | `analysis_engine.rs` emit `pipeline-progress` 事件 |
| D6 | V1 老 Rust 命令保留（后续清理）|

新增事件协议 `pipeline-progress`：`{ phase: 1-4, status: "started"|"completed"|"failed", message, progress }`

### 任务四a：library Tab DB 化 + 多维检索

**后端**：`db.rs` 新增 `list_novels(filter)` 查询：
- 支持 tags/consensus OR 过滤 + platform 等于 + 三种排序
- SQL 层用子查询拿 `latest_rank` + `scan_count`
- `ai_reviews_json` 在 Rust 层 parse（非原始字符串）
- 参数化 LIKE 防 SQL 注入
- 4 个单元测试覆盖默认全取 / tag OR / consensus 过滤 / tags CSV 解析

**前端**：`NovelCard.vue` 组件首次提取：
- Props/Emits 类型化，`consensus` 五枚举徽章配色（绿/翠/琥珀/玫瑰/红）
- vote 三值映射 emoji（👍/👎/🤔）
- information density = 中等（标题 + tags + consensus + vote icon + rank + 字数 + 上榜次数）

### 任务四b：趋势洞察面板（方向修正）

**原始需求**是「雷达图」，用户指出**真实需求**是：「快速知道近期可写什么题材、哪些设定读者喜欢」。

| 共识分布 | `consensusDistribution`: 五枚举 + unrated 计数 |

## 架构变化：V1 → V2

### V1 单体架构
```
用户输入 URL → start_download / scan_and_download_rank
  → 串行下载章节 → 写文件到 downloads/ 目录
  → 可选 AI 分析 → 写 info.json
```

### V2 流水线架构
```
Scheduler（定时触发） 或  前端 trigger_full_scan（手动触发）
  → Phase 1 Producer: 扫榜分发，写 DB
  → Phase 2 Fetch: Semaphore(3) 并发抓取，写 DB
  → Phase 3 AI Outline: Semaphore(3) 并发提纯，写 DB
  → Phase 4 Multi-Agent: 三 Agent 并行评估，写 DB
  → 生成报告 + emit pipeline-progress 事件
  → 前端自动刷新
```

### 数据流向变化
| | V1 | V2 |
|---|---|---|
| 存储 | `downloads/` 文件目录 | SQLite（novels + chapters + scan_reports + rank_history 4 表） |
| 元数据 | `info.json` | `novels.ai_reviews_json`（含多 Agent 评估）|
| 章节 | `.txt` 文件 | `chapters.content` + `chapters.outline_json`（AI 细纲）|
| 前端数据源 | `get_file_tree`（读文件夹） | `list_novels`（查询 DB） |
